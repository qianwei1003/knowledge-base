# 记忆系统

## 概述

记忆系统是 Agent 的核心能力组件，负责存储和检索上下文信息，使 Agent 能够"记住"之前的对话和知识。

---

## 记忆类型

### 1. 短期记忆（工作记忆）
| 特征 | 说明 |
|------|------|
| 容量 | 有限（受Token限制） |
| 存活期 | 当前会话 |
| 内容 | 当前对话上下文 |
| 实现 | LLM上下文窗口 |

### 2. 长期记忆
| 特征 | 说明 |
|------|------|
| 容量 | 可扩展 |
| 存活期 | 永久（或用户删除） |
| 内容 | 历史对话、用户偏好、学习到的知识 |
| 实现 | 向量数据库、关系数据库 |

### 3. 知识库记忆
| 特征 | 说明 |
|------|------|
| 容量 | 可扩展 |
| 存活期 | 永久（或更新） |
| 内容 | 领域知识、规范文档、操作手册 |
| 实现 | RAG检索、知识图谱 |

## 关键设计

### 1. 写入策略
- **全量写入**：每条消息都存储
- **摘要写入**：只存储摘要，节省空间
- **选择性写入**：只存储关键信息

### 2. 检索策略
- **最近N条**：取最近的N条消息
- **语义检索**：按相关性检索（向量相似度）
- **混合检索**：关键词 + 语义

### 3. 淘汰策略
- **FIFO**：先进先出
- **重要性淘汰**：保留重要的，删除次要的
- **摘要压缩**：将旧内容压缩为摘要

## 实现方案

### 方案1：纯上下文（简单）
```
直接将历史对话作为上下文传入LLM

优点：实现简单
缺点：上下文长度受限，成本高
```

### 方案2：向量检索（推荐）
```
1. 将历史对话向量化存储
2. 检索时按相似度召回
3. 将召回内容加入上下文

优点：可扩展，检索精准
缺点：需要向量数据库
```

### 方案3：摘要压缩
```
1. 定期将历史对话压缩为摘要
2. 只保留摘要 + 最近几轮对话

优点：节省空间
缺点：可能丢失细节
```

### 方案4：分层记忆（完整）
```
短期：最近N轮对话
中期：近期重要信息的摘要
长期：向量数据库存储的历史
知识：RAG检索的领域知识

综合检索：短期 + 中期检索 + 长期检索 + 知识检索
```

## 记忆内容格式

### 对话记忆
```json
{
  "role": "user|assistant",
  "content": "消息内容",
  "timestamp": "2026-04-18T12:00:00Z",
  "importance": 0.8,
  "tags": ["主题1", "主题2"]
}
```

### 摘要记忆
```json
{
  "type": "summary",
  "period": "2026-04-01 ~ 2026-04-18",
  "key_points": ["要点1", "要点2"],
  "decisions": ["决策1", "决策2"],
  "created_at": "2026-04-18"
}
```

### 知识记忆
```json
{
  "source": "知识库路径",
  "content": "知识内容",
  "embedding": [0.1, 0.2, ...],
  "metadata": {
    "category": "分类",
    "updated_at": "2026-04-18"
  }
}
```

## 最佳实践

### 1. 记忆分级
- 重要性 > 0.8：永久保存
- 重要性 0.5-0.8：保留30天
- 重要性 < 0.5：临时存储

### 2. 隐私处理
- 敏感信息不存储
- 用户可删除记忆
- 定期清理过期记忆

### 3. 性能优化
- 批量写入，减少IO
- 异步检索，不阻塞主流程
- 缓存热门记忆

## 常见问题

| 问题 | 解决方案 |
|------|---------|
| 上下文太长 | 摘要压缩 + 向量检索 |
| 检索不准 | 调整embedding模型 + 关键词增强 |
| 记忆冲突 | 时间戳 + 来源标记 |
| 记忆泄露 | 权限控制 + 隔离存储 |
