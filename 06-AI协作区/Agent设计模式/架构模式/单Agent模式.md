# 单Agent模式

## 适用场景

- 单一明确的任务目标
- 任务流程相对固定
- 不需要复杂分工
- 用户直接交互

---

```
┌─────────────────────────────────────────┐
│              单Agent 架构                │
├─────────────────────────────────────────┤
```

---

## 核心组件

### 1. Agent 核心
- **职责**：接收请求、推理决策、生成输出
- **关键参数**：模型选择、Temperature、最大Token

### 2. 记忆系统
- **短期记忆**：当前对话上下文
- **长期记忆**：历史对话、用户偏好
- **知识库**：领域知识、规范文档

### 3. 工具调用
- **内置工具**：搜索、计算、代码执行
- **外部工具**：API调用、文件操作

### 4. 输出格式
- **文本**：自然语言回复
- **结构化**：JSON、Markdown、代码

---

## 实现要点

### 1. 明确边界
```
Agent 名称：xxxAgent
职责：[一句话描述]
输入：[用户可以给什么]
输出：[Agent 会返回什么]
边界：[Agent 不做什么]
```

### 2. 系统提示词设计
```
你是 [Agent名称]，专门负责 [职责]。

## 你的能力
- 能力1
- 能力2

## 你的边界
- 不做xxx
- 遇到xxx时，告知用户

## 输出格式
[格式规范]

## 注意事项
[关键约束]
```

### 3. 错误处理
- 输入不符合预期时，给出明确提示
- 无法完成任务时，说明原因
- 关键操作需用户确认

---

## 优缺点

| 优点 | 缺点 |
|------|------|
| 结构简单，易于实现 | 复杂任务能力有限 |
| 调试方便 | 单点故障风险 |
| 用户交互直接 | 无法并行处理 |
| 资源消耗低 | 扩展性受限 |

## 典型案例

### 案例：代码解释 Agent
```
名称：CodeExplainer
职责：解释代码逻辑、发现潜在问题

输入：
- 代码片段
- 用户问题（可选）

输出：
- 代码功能解释
- 关键逻辑说明
- 潜在问题提示

能力：
- 代码分析
- 文档查询

边界：
- 不执行代码
- 不修改代码
```

### 案例：知识问答 Agent
```
名称：KnowledgeQA
职责：基于知识库回答问题

输入：
- 用户问题

输出：
- 答案
- 来源引用

能力：
- 知识库检索
- 上下文理解

边界：
- 只回答知识库范围内的问题
- 不确定时明确告知
```

## 最佳实践

1. **从单Agent开始**：不要一开始就设计复杂的多Agent系统
2. **边界要清晰**：明确告诉用户 Agent 能做什么、不能做什么
3. **失败可解释**：Agent 做不到时，要能解释原因
4. **渐进增强**：先保证基本功能，再添加高级能力
