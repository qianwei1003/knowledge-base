# 微服务架构指南

> 微服务架构风格是一种将单个应用程序开发为一套小型服务的方法，每个服务运行在自己的进程中，通过轻量级机制（通常是 HTTP 资源 API）进行通信。这些服务围绕业务能力构建，可通过全自动部署机制独立部署。
>
> — James Lewis & Martin Fowler (2014)

## 微服务的核心特征

### 1. 通过服务实现组件化

微服务将应用程序分解为多个服务，每个服务作为一个独立的组件。组件是可独立替换和升级的软件单元。与库（在进程内运行）不同，服务是进程外组件，通过明确的远程接口（如 HTTP API）通信。

**关键优势：** 服务可以独立部署，不需要整个应用一起重新发布。

### 2. 围绕业务能力组织

微服务按业务领域而非技术功能来组织团队和代码。每个服务对应一个业务能力边界（领域驱动设计中的"限界上下文"）。

**示例：** 电商系统中 → 用户服务、订单服务、库存服务、支付服务，而非前端服务、后端服务、数据库服务。

### 3. 产品而非项目

团队不应该把服务当成一个需要完成然后交接给运维的项目，而应该把每个服务当作一个**产品**来全生命周期负责，从设计、开发、测试到部署、监控、维护。

### 4. 智能端点与哑管道

微服务倾向于"智能端点、哑管道"的架构：业务逻辑放在服务端点中，通信管道保持简单（如 REST API、消息队列），避免在中间件中放置业务逻辑。

**推荐通信方式：** RESTful API、gRPC、消息队列（如 Kafka、RabbitMQ）
**不推荐：** 复杂的 ESB（企业服务总线）编排

### 5. 去中心化治理

微服务推崇技术多样性：不同服务可以使用不同的编程语言、框架和数据库技术。选择技术时应以"最适合这个服务的需求"为标准，而非"全公司统一"。

### 6. 去中心化数据管理

每个微服务拥有自己独立的数据存储，**不共享数据库**。这是微服务架构的基本原则之一。

**为什么重要：**
- 避免数据库层面的耦合
- 服务可以独立演进其数据模型
- 不同服务可以选择最适合的数据存储技术

### 7. 基础设施自动化

微服务的成功部署严重依赖自动化：
- **CI/CD 管道：** Jenkins、GitHub Actions、GitLab CI
- **容器化：** Docker
- **编排：** Kubernetes
- **基础设施即代码：** Terraform、Ansible

### 8. 容错设计（Design for Failure）

分布式系统中，任何服务都可能失败。微服务架构必须：

- 使用**断路器模式**（Circuit Breaker）防止级联故障
- 实现**超时机制**和**重试策略**
- 提供**优雅降级**方案
- 设计**健康检查**接口

**断路器模式要点：**
1. 监控远程调用的失败次数
2. 达到阈值时"断开"电路，直接返回错误（不再发起远程调用）
3. 一段时间后尝试半开状态，检测服务是否恢复
4. 恢复正常后关闭断路器

### 9. 演进式设计

微服务架构不是一次性设计出来的，而是在实践中逐步演进的。重要的是能快速响应变化，通过不断重构和拆分来保持架构的合理性。

## 微服务的优势

| 优势 | 说明 |
|------|------|
| **强模块边界** | 强化模块化结构，对大型团队尤其重要 |
| **独立部署** | 简单服务更容易部署，出错影响范围小 |
| **技术多样性** | 可混用多种语言、框架和数据存储技术 |
| **团队自治** | 小团队可以独立开发和部署自己的服务 |
| **弹性扩展** | 可对高负载服务单独扩容 |
| **故障隔离** | 单个服务故障不会拖垮整个系统 |

## 微服务的代价

| 代价 | 说明 |
|------|------|
| **分布式复杂性** | 远程调用慢且容易失败，需要处理网络超时、重试、幂等性 |
| **最终一致性** | 分布式系统难以保持强一致性，必须管理最终一致性 |
| **运维复杂度** | 需要成熟的运维团队管理大量服务 |
| **调试困难** | 跨服务调用链的追踪和调试比单体应用更复杂 |
| **数据一致性** | 跨服务事务处理需要 Saga 模式等复杂方案 |
| **测试复杂度** | 需要契约测试、集成测试等更多测试策略 |

## 何时使用微服务？

### ✅ 适合使用的情况

- 应用已经很大且不断增长
- 团队规模大到需要拆分
- 不同模块有不同的扩展需求
- 不同模块有不同的技术需求
- 组织已有成熟的 DevOps 能力

### ❌ 不适合使用的情况

- 小型应用或初创项目
- 团队不具备 DevOps 成熟度
- 应用复杂度不够高
- 没有足够的基础设施支持

> **Monolith First 原则：** 几乎所有成功的微服务故事都是从单体应用开始，当它变得太大时才拆分。从零开始就构建微服务系统通常会导致严重问题。
>
> — Martin Fowler

## 微服务拆分原则

### 单一职责原则
每个微服务专注于一个特定的业务功能，避免构建会因多个原因而变更的"臃肿服务"。

### 数据库独立
所有微服务不应共享一个数据库。每个服务应拥有自己的数据存储，其他服务只能通过 API 访问数据。

### 异步通信实现松耦合
在微服务之间使用异步通信（事件/消息），避免构建紧耦合的服务网格：
- 服务发布事件到消息总线
- 对该事件感兴趣的服务订阅并处理
- 减少直接的服务间依赖

### API 网关
通过统一的 API 网关代理微服务请求，集中处理：
- 身份认证和授权
- 请求限流
- 日志记录
- 路由和负载均衡

### 版本兼容
- 确保变更向后兼容
- 使用合约测试（Consumer-Driven Contract Testing）
- 重大变更时提供新版本接口并逐步废弃旧版本

## 微服务测试策略

| 测试类型 | 范围 | 说明 |
|---------|------|------|
| 单元测试 | 单个服务内 | 测试服务内部逻辑 |
| 契约测试 | 服务间接口 | 验证 API 兼容性 |
| 集成测试 | 服务+基础设施 | 测试与数据库、消息队列的交互 |
| 端到端测试 | 完整系统 | 测试跨服务的业务流程 |

## 从单体迁移到微服务

1. **从简单开始**：先抽取一个最简单的服务
2. **选择垂直能力**：选择对业务重要且频繁变更的模块
3. **服务先大后小**：初期服务可以大一些，不宜过细
4. **避免依赖单体**：新服务应尽量不依赖剩余的单体系统
5. **每步都是改进**：确保每次迁移步骤都是对整体架构的原子性改进
6. **保持数据单一写源**：拆分数据时始终只保留一份写拷贝
