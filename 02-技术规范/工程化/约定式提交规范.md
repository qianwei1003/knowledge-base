# 约定式提交规范 (Conventional Commits v1.0.0)

## 核心要点

- **三要素**：类型 + 可选范围 + 描述，格式 `<type>[scope]: <description>`
- **SemVer 映射**：`feat` → MINOR，`fix` → PATCH，`BREAKING CHANGE` → MAJOR
- **破坏性变更标记**：脚注 `BREAKING CHANGE:` 或类型后加 `!`（如 `feat!:`）
- **自动化价值**：可自动生成 CHANGELOG、决定版本号、触发构建部署
- **正文与脚注**：正文用空行与描述分隔，脚注用 `-` 连字符（如 `Reviewed-by`）

---

## 提交信息格式

```
<类型>[可选 范围]: <描述>

[可选 正文]

[可选 脚注]
```

## 核心类型

| 类型 | 说明 | SemVer |
|------|------|:------:|
| `feat` | 新增功能 | MINOR |
| `fix` | 修复 Bug | PATCH |
| `BREAKING CHANGE` | 破坏性 API 变更 | MAJOR |

## 扩展类型（Angular 约定）

| 类型 | 说明 |
|------|------|
| `build` | 构建系统/依赖变更 |
| `chore` | 非业务性修改 |
| `ci` | CI 配置修改 |
| `docs` | 文档修改 |
| `style` | 代码样式（不影响逻辑） |
| `refactor` | 重构（不修改功能） |
| `perf` | 性能优化 |
| `test` | 测试用例 |

## 示例

```bash
# 基本
docs: correct spelling of CHANGELOG
feat(lang): add polish language

# 破坏性变更 — 脚注形式
feat: allow provided config object to extend other configs

BREAKING CHANGE: `extends` key in config file is now used for extending other config files

# 破坏性变更 — ! 标记
feat!: send an email to the customer when a product is shipped
feat(api)!: send an email to the customer when a product is shipped

# 完整示例
fix: prevent racing of requests

Introduce a request id and a reference to latest request. Dismiss
incoming responses other than from latest request.

Reviewed-by: Z
Refs: #123
```

## 强制规则

1. ✅ 每个提交**必须**使用类型前缀
2. ✅ 新功能 `feat`，修 bug `fix`，不可混用
3. ✅ 范围用圆括号：`fix(parser):`
4. ✅ 描述直接跟冒号+空格
5. ✅ `BREAKING CHANGE` **必须大写**
6. ❌ 反模式：`update code`、`fix bug` 等无类型前缀的提交
7. ❌ 反模式：描述与正文之间无空行
