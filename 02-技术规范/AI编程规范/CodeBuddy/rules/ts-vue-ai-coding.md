# TypeScript + Vue AI 编程规范

> 基于《通病与陷阱》和 ECC AI Engineering 最佳实践
> 来源：[[02-技术规范/AI编程规范/通病与陷阱]]、[[02-技术规范/AI编程规范/ECC-AI编程与自主循环模式]]

---

## 核心原则

### AI-First Engineering
- **规划质量 > 打字速度**：先理解需求再动手
- **评估覆盖率 > 主观信心**：用验证代替猜测
- **15分钟单元规则**：每个任务可独立验证、有清晰完成条件

### 编码哲学
- **KISS**：简单 > 巧妙 > 复杂
- **DRY**：不要重复自己
- **YAGNI**：不要实现不需要的功能

---

## AI 行为约束

### 禁止敷衍式代码生成
- **错误行为**：快速扫描需求后立即编码，只关注功能实现，不做自我审查
- **正确做法**：深入理解需求，生成代码后逐行审查，以“能过 code review”为标准
- **强制执行**：必须执行 ECC 六阶段验证

### 禁止敷衍式代码审查
- **错误行为**：快速扫描结构，给出模板化“合规”结论，不发现具体问题
- **正确做法**：逐行分析，提供行号级别问题报告，对照《通病与陷阱》逐项核对
- **强制执行**：审查报告必须包含具体行号问题，不能只给整体评价

## 代码质量约束

### 编码规范
1. **命名规范**：变量 camelCase，常量 UPPER_SNAKE_CASE，类/接口 PascalCase，布尔值 is/has/should 前缀
2. **禁止魔法值**：所有字面量字符串/数字必须提取为常量（包括错误消息、事件名、存储 key、API 路径等）
3. **禁止硬编码**：配置值、敏感信息必须使用环境变量
4. **禁止过度设计**：简单功能不用设计模式，一个类/函数只被使用一次则简化
5. **禁止遗留 TODO**：拆分任务完整实现，用 issue 跟踪后续工作

### 错误处理
1. **异步操作必须 try-catch**：所有 `await` 和 `JSON.parse` 必须包裹错误处理
2. **空值检查**：访问对象属性或数组元素前必须验证存在性（使用 `?.`、`??`）
3. **具体异常捕获**：禁止裸 `catch (e) {}`，必须捕获具体异常类型并处理

### 类型安全
1. **禁止随意使用 any**：使用 `unknown` + 类型守卫替代，特殊情况需注释理由
2. **禁止滥用类型断言**：优先使用类型守卫，类型断言是最后手段
3. **优先使用 interface**：定义对象类型时优先 `interface` 而非 `type`

### Vue 3 组合式 API
1. **强制使用 script setup**：必须使用 `<script setup>` 语法
2. **Props 必须类型声明**：使用泛型类型声明，禁止数组格式
3. **Emits 必须声明**：使用 `defineEmits` 声明类型，禁止直接 emit

### 原生 API 封装
1. **fetch 必须封装**：在 `useApi` 或类似 composable 中封装，禁止在业务组件中直接使用
2. **localStorage/sessionStorage 必须封装**：使用 `useStorage` composable 封装
3. **第三方 SDK/API 必须抽象适配层**：禁止在业务代码中直接调用

### 组件设计
1. **单一职责**：一个组件只做一件事
2. **行数限制**：组件超过 150 行必须考虑拆分
3. **可复用组件**：可复用的 UI 元素必须组件化（BaseInput、BaseButton 等）
4. **避免重复代码**：相同逻辑出现 3 次以上 → 提取公共函数

### 性能优化
1. **避免循环内重复计算**：循环外计算结果缓存
2. **避免不必要内存分配**：简化中间变量，直接计算

## 安全约束

### 输入验证
- **所有函数参数必须验证**：使用 Zod 等库验证类型、格式、范围
- **用户输入必须转义**：禁止直接输出未转义的用户输入

### 防止注入
- **SQL 注入**：永远使用参数化查询，禁止字符串拼接 SQL
- **XSS**：Vue 默认转义，必须使用 `v-html` 时先净化（如 DOMPurify）

### 敏感信息
- **禁止硬编码密钥**：API 密钥、密码等必须使用环境变量
- **提交前扫描**：每次提交前扫描 `sk-`、`password`、`secret` 等关键词

## 测试与文档

### 测试要求
1. **覆盖率 ≥80%**：单元测试必须达到 80% 覆盖率
2. **测试场景**：正常输入、边界值、异常输入、并发情况（如有）
3. **测试行为而非实现**：测试用户可见行为，而非内部状态

### 文档要求
1. **公共 API 必须文档注释**：所有导出函数必须有 JSDoc 注释
2. **内容必须**：参数说明、返回值、异常、使用示例
3. **格式规范**：使用规范格式，包含 `@param`、`@returns`、`@throws`、`@example`

## ECC 验证循环（强制）

每次代码产出后必须执行以下验证，不通过不得提交：

```bash
Phase 1: 构建验证    → npm run build
Phase 2: 类型检查    → npx tsc --noEmit
Phase 3: Lint 检查   → npm run lint
Phase 4: 测试套件    → npm run test -- --coverage (目标80%)
Phase 5: 安全扫描    → 检查 sk-、console.log、敏感信息
Phase 6: Diff 审查   → git diff --stat 审查每个变更
```

**验证报告格式**：
```
VERIFICATION REPORT
==================
构建:     [PASS/FAIL]
类型:     [PASS/FAIL] (X errors)
Lint:     [PASS/FAIL] (X warnings)
测试:     [PASS/FAIL] (X/Y passed, Z% coverage)
安全:     [PASS/FAIL] (X issues)
Diff:     [X files changed]

Overall:  [READY/NOT READY] for PR

Issues to Fix:
1. ...
```

## 执行检查清单

### 生成代码前
- [ ] 深入理解需求（不是扫描关键词）
- [ ] 规划实现方案（不是立即编码）

### 生成代码时
- [ ] 无硬编码配置值/敏感信息
- [ ] 原生 API（fetch/localStorage）已封装
- [ ] 无 TODO/FIXME/NotImplementedError
- [ ] 异步操作有 try-catch
- [ ] 有空值检查（`?.`、`??`）
- [ ] 异常捕获具体类型
- [ ] 用户输入已验证（Zod）
- [ ] 无 SQL 注入/XSS 风险
- [ ] 无随意使用 any
- [ ] 优先使用 interface
- [ ] 使用 script setup
- [ ] Props/Emits 有类型声明
- [ ] 命名规范
- [ ] 无魔法数字/字符串
- [ ] 组件不超过 150 行
- [ ] 可复用 UI 已组件化

### 生成代码后
- [ ] 逐行阅读代码（不是扫描）
- [ ] 检查变量/函数命名是否清晰
- [ ] 检查是否有重复代码可提取
- [ ] 检查组件是否过度臃肿
- [ ] 对照《通病与陷阱》逐一核对
- [ ] 问自己：这代码能过 review 吗？

### 测试与文档
- [ ] 测试覆盖率 ≥80%
- [ ] 公共 API 有文档注释
- [ ] ECC 六阶段验证通过
- [ ] 审查报告包含行号级具体问题
