# AI 编程通病与陷阱

> 记录 AI 在编程过程中常见的问题和错误，作为规范制定的重要信息来源。

---

## 一、代码生成类问题

### 1.1 硬编码配置值

**问题描述：** AI 倾向于在代码中直接硬编码配置值，而非使用配置文件或环境变量。

**典型表现：**
```javascript
// ❌ 错误示例
const API_URL = "https://api.example.com/v1";
const MAX_RETRIES = 3;
const TIMEOUT_MS = 5000;

// ✅ 正确示例
const API_URL = process.env.API_URL;
const MAX_RETRIES = parseInt(process.env.MAX_RETRIES || '3', 10);
```

**预防措施：**
- 在提示词中明确要求使用配置文件
- 添加配置验证逻辑

**来源：** [[通病与陷阱]] #AI生成

---

### 1.2 硬编码敏感信息

**问题描述：** AI 可能生成包含 API Key、密码、密钥等敏感信息的代码。

**典型表现：**
```python
# ❌ 错误示例
API_KEY = "sk-1234567890abcdef"
PASSWORD = "admin123"

# ✅ 正确示例
import os
API_KEY = os.environ.get('API_KEY')
PASSWORD = os.environ.get('ADMIN_PASSWORD')
```

**预防措施：**
- 始终在提示词中声明不使用硬编码敏感信息
- 启用代码安全扫描

---

### 1.3 过度复杂的设计

**问题描述：** AI 倾向于使用过度设计的设计模式，导致代码复杂度不必要地增加。

**典型表现：**
- 为简单功能创建工厂模式
- 使用多层抽象处理一次性任务
- 过度使用泛型/模板

**判断标准：**
- 如果一个类/函数只被使用一次，考虑简化
- KISS 原则：简单 > 巧妙 > 复杂

---

### 1.4 不完整的实现

**问题描述：** AI 可能只实现部分功能，留下 TODO 注释或抛出 `NotImplementedError`。

**典型表现：**
```python
def complex_function(data):
    # TODO: implement validation
    # TODO: add error handling
    # FIXME: this might not work for edge cases
    pass
```

**预防措施：**
- 明确要求完整实现
- 拆分复杂任务为多个简单任务

## 二、错误处理类问题

### 2.1 缺失错误处理

**问题描述：** AI 生成的代码往往缺少适当的异常处理和边界情况处理。

**典型表现：**
```javascript
// ❌ 错误示例
const data = JSON.parse(userInput);

// ✅ 正确示例
let data;
try {
  data = JSON.parse(userInput);
} catch (error) {
  logger.error('Invalid JSON input:', error);
  return { error: 'Invalid input format' };
}
```

---

### 2.2 空值/null 检查缺失

**问题描述：** AI 经常忽略空值检查，导致运行时错误。

**典型表现：**
```python
# ❌ 错误示例
name = user.profile.name  # 如果 profile 或 name 为 None

# ✅ 正确示例
name = user.profile.name if user.profile else "Anonymous"
# 或使用 Optional 类型和匹配
```

---

### 2.3 过于宽泛的异常捕获

**问题描述：** 使用 `except Exception` 或 `catch (...)` 捕获所有异常。

**典型表现：**
```python
# ❌ 错误示例
try:
    do_something()
except Exception:
    pass

# ✅ 正确示例
try:
    do_something()
except ValueError as e:
    logger.error(f'Invalid value: {e}')
except ConnectionError as e:
    logger.error(f'Connection failed: {e}')
```

## 三、安全类问题

### 3.1 SQL 注入风险

**问题描述：** 使用字符串拼接构建 SQL 查询。

**典型表现：**
```python
# ❌ 错误示例
query = f"SELECT * FROM users WHERE name = '{username}'"

# ✅ 正确示例
query = "SELECT * FROM users WHERE name = %s"
cursor.execute(query, (username,))
```

---

### 3.2 XSS 漏洞

**问题描述：** 直接输出用户输入而不进行转义。

**典型表现：**
```html
<!-- ❌ 错误示例 -->
<div>{{ userInput }}</div>

<!-- ✅ 正确示例 -->
<div>{{ escape(userInput) }}</div>
<!-- 或使用框架的自动转义 -->
```

---

### 3.3 缺乏输入验证

**问题描述：** 不验证用户输入的类型、格式和范围。

**典型表现：**
```typescript
// ❌ 错误示例
function calculateAge(birthYear) {
  return 2024 - birthYear;
}

// ✅ 正确示例
function calculateAge(birthYear: number): number {
  if (birthYear < 1900 || birthYear > 2024) {
    throw new ValidationError('Invalid birth year');
  }
  return new Date().getFullYear() - birthYear;
}
```

## 四、代码质量类问题

### 4.1 命名不规范

**问题描述：** 变量、函数命名不符合语言习惯或团队规范。

**典型表现：**
```javascript
// ❌ 不规范命名
let dataArray = [];
let tempVar = getSomething();
let result2 = process(data2);

// ✅ 规范命名
let userList = [];
let temporaryValue = getUserInput();
let processedResult = process(rawData);
```

---

### 4.2 注释缺失或误导

**问题描述：** 缺少必要注释，或注释与代码不一致。

**典型表现：**
```javascript
// ❌ 误导性注释
// 递归计算阶乘
function factorial(n) {
  return n * factorial(n - 1) + 1; // 错误实现但注释说正确
}

// ✅ 正确注释
/**
 * 计算阶乘
 * @param n 非负整数
 * @returns n!
 * @throws {RangeError} 当 n 为负数时
 */
function factorial(n) {
  if (n < 0) throw new RangeError('n must be non-negative');
  return n <= 1 ? 1 : n * factorial(n - 1);
}
```

---

### 4.3 魔法数字/字符串

**问题描述：** 在代码中直接使用未命名的数字或字符串常量。

**典型表现：**
```python
# ❌ 魔法数字
if user.age > 18 and user.score > 60:

# ✅ 命名常量
ADULT_AGE = 18
PASSING_SCORE = 60

if user.age > ADULT_AGE and user.score > PASSING_SCORE:
```

---

### 4.4 重复代码

**问题描述：** AI 容易在多次对话中生成相似的代码块，而非提取公共函数。

**预防措施：**
- 定期要求 AI 进行代码重构
- 使用 DRY 原则检查

---

## 五、性能类问题

### 5.1 循环内的重复计算

**典型表现：**
```python
# ❌ 错误示例
for item in items:
    result = expensive_calculation()  # 每次循环都计算
    process(item, result)

# ✅ 正确示例
cached_result = expensive_calculation()  # 循环外计算一次
for item in items:
    process(item, cached_result)
```

---

### 5.2 不必要的内存分配

**典型表现：**
```javascript
// ❌ 错误示例
const result = data.map(x => {
  const temp = compute(x);
  return temp * 2;
});

// ✅ 正确示例
const result = data.map(x => compute(x) * 2);
```

## 六、类型系统类问题

### 6.1 忽视类型安全

**典型表现（TypeScript）：**
```typescript
// ❌ 错误示例
function processData(data: any) {
  return data.value.name;
}

// ✅ 正确示例
interface User {
  value: {
    name: string;
  };
}

function processData(data: User) {
  return data.value.name;
}
```

---

### 6.2 类型断言滥用

**典型表现：**
```typescript
// ❌ 滥用 as
const user = getData() as User;

// ✅ 正确做法
const user = getData();
if (isUser(user)) {
  processUser(user);
}
```

---

## 七、测试类问题

### 7.1 测试覆盖率不足

**典型表现：**
- 只测试 happy path
- 不测试边界条件
- 不测试异常情况

**建议测试场景：**
1. 正常输入
2. 边界值（0、最大值、空值）
3. 异常输入
4. 并发情况（如果有）

---

### 7.2 测试与实现耦合

**典型表现：**
```python
# ❌ 测试内部实现
def test_internal_logic(self):
    assert obj._cache.get('key') == value

# ✅ 测试外部行为
def test_returns_cached_value(self):
    result = obj.get_value('key')
    assert result == expected_value
```

## 八、文档类问题

### 8.1 API 文档缺失

**典型表现：**
- 函数没有文档注释
- 参数含义不明确
- 返回值格式未说明

**建议格式：**
```typescript
/**
 * 获取用户信息
 * 
 * @param userId - 用户 ID，支持 UUID 格式
 * @param options - 可选配置
 * @param options.includeProfile - 是否包含详细资料，默认 false
 * @returns 用户信息对象，包含 id、name、email 字段
 * @throws {NotFoundError} 当用户不存在时
 * @throws {ValidationError} 当 userId 格式无效时
 * 
 * @example
 * const user = await getUser('123', { includeProfile: true });
 */
async function getUser(userId: string, options?: { includeProfile?: boolean }): Promise<User> {
  // ...
}
```
