# SDD 规范驱动开发

> 收集整理 AI 时代的规范驱动开发（Spec-Driven Development, SDD）最佳实践，让 AI 从"自由发挥的对话者"转变为"遵循规范的协作者"。

---

## 一、核心概念

### 1.1 什么是 SDD

**SDD（Spec-Driven Development，规范驱动开发）** 是一种将规范作为核心交付物的开发方法论。它通过结构化的规范文档来约束 AI 行为，确保需求、设计、代码的一致性。

### 1.2 SDD 核心问题

| 问题 | 描述 | 影响 |
|-----|------|------|
| **上下文丢失** | AI 难以保持长对话中的上下文一致性 | 代码片段不连贯 |
| **需求模糊** | 需求定义不清晰导致代码偏离预期 | 反复返工 |
| **质量不稳定** | AI 输出质量依赖模型能力 | 难以保证下限 |

### 1.3 SDD 解决方案

```
传统模式 → SDD新范式

❌ AI 自由生成代码   →  ✅ AI 遵循结构化规范
❌ 需求口头传达      →  ✅ 需求文档化、可追溯
❌ 代码质量依赖AI能力 →  ✅ 代码变为可验证的单元
❌ 协作依赖人工review →  ✅ 规范驱动自动一致性
```

## 二、OpenSpec 框架

### 2.1 核心工作流

```
┌─────────────────────────────────────────────────────────────┐
│                    SDD 四阶段工作流                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐   │
│   │Proposal  │ →  │   Spec   │ →  │  Design  │ →  │  Tasks   │   │
│   │ 变更提案  │    │ 规范定义  │    │ 方案设计  │    │ 任务拆解  │   │
│   └──────────┘    └──────────┘    └──────────┘    └──────────┘   │
│        ↓              ↓              ↓              ↓           │
│    变更需求       规范化文档       设计方案       可执行任务       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 各阶段说明

#### Proposal（变更提案）

| 要素 | 说明 |
|-----|------|
| **目的** | 明确要解决什么问题 |
| **输入** | 业务需求、用户反馈、技术债务 |
| **输出** | 变更提案文档 |

**提案模板：**

```markdown
# 变更提案：[变更名称]

## 问题背景
[描述当前存在的问题]

## 变更目标
- [目标1]
- [目标2]

## 约束条件
- [约束1]
- [约束2]

## 预期收益
- [收益1]
- [收益2]
```

#### Spec（规范定义）

| 要素 | 说明 |
|-----|------|
| **目的** | 定义"做什么"和"怎么做" |
| **输入** | Proposal、现有规范 |
| **输出** | 规范文档（Spec） |

**规范模板：**

```markdown
# 规范文档：[功能名称]

## 功能概述
[简述功能目标和范围]

## 功能需求
### F1：用户登录
- **优先级**：P0
- **前置条件**：用户已注册
- **正常流程**：
  1. 用户输入用户名和密码
  2. 系统验证凭证
  3. 验证成功则跳转首页
- **异常流程**：
  1. 用户名不存在 → 提示"用户名不存在"
  2. 密码错误 → 提示"密码错误"

## 接口规范
### POST /api/login
| 参数 | 类型 | 必填 | 说明 |
|-----|------|------|------|
| username | string | 是 | 用户名 |
| password | string | 是 | 密码 |

## 验收标准
- [ ] 用户名不存在时提示正确
- [ ] 密码错误时提示正确
- [ ] 登录成功后跳转到首页
```

#### Design（方案设计）

| 要素 | 说明 |
|-----|------|
| **目的** | 定义技术实现方案 |
| **输入** | Spec、技术选型 |
| **输出** | 设计文档 |

**设计模板：**

```markdown
# 技术设计文档：[功能名称]

## 技术选型
| 组件 | 选择 | 理由 |
|-----|------|------|
| 前端框架 | React 18 | 团队熟悉度 |
| 状态管理 | Zustand | 轻量级 |

## 模块设计
### UserLogin 模块
```typescript
interface LoginForm {
  username: string;
  password: string;
}

function useLogin(): {
  login: (form: LoginForm) => Promise<void>;
  error: string | null;
  loading: boolean;
}
```

#### Tasks（任务拆解）

| 要素 | 说明 |
|-----|------|
| **目的** | 将设计转化为可执行任务 |
| **输入** | Design |
| **输出** | 任务列表 |

**任务模板：**

```markdown
# 任务列表：[功能名称]

## 开发任务
- [ ] T1: 创建 LoginForm 组件
- [ ] T2: 实现 useLogin hook
- [ ] T3: 接入登录 API
- [ ] T4: 添加表单验证
- [ ] T5: 编写单元测试

## 验收任务
- [ ] T6: 集成测试
- [ ] T7: E2E 测试
- [ ] T8: 文档更新
```

## 三、AI 时代的 SDD 应用

### 3.1 让 AI 成为规范遵守者

| 传统 AI 使用 | SDD 模式 |
|-------------|---------|
| "帮我写一个登录功能" | 提供完整的 Spec + Design |
| AI 自由发挥 | AI 按规范执行 |
| 质量依赖 AI 能力 | 质量由规范保证 |

### 3.2 SDD + AI 提示词模板

```markdown
# SDD 模式 AI 编程提示词

## 上下文
你正在开发一个{项目名称}项目。

## 当前阶段
[ ] Proposal  [x] Spec  [ ] Design  [ ] Tasks

## 项目规范
请严格遵循以下规范：
1. 大厂编程规范
2. Airbnb JS规范摘要

## 当前任务
{粘贴对应的 Spec/Design/Tasks 内容}

## 执行要求
1. 严格按照规范执行，不要偏离
2. 如需变更，先更新规范文档
3. 每个产出都要标注对应规范条款
4. 完成后自审是否符合规范

## 验收标准
{粘贴验收标准}
```

### 3.3 规范文档示例

#### 变更即代码（Change as Code）

```markdown
# 变更即代码模式

## 核心思想
将变更定义为可追溯、可验证的代码单元，而非口头指令。

## 实施方式

### 1. 变更规格化
每个变更都包含：
- **变更ID**：唯一标识（如 CHANGE-2026-001）
- **变更描述**：清晰说明变更内容
- **影响范围**：涉及的模块和文件
- **验收标准**：可验证的检查项

### 2. 变更即 PR
```markdown
## PR: CHANGE-2026-001 - 添加用户登录功能

### 变更规格
- ID: CHANGE-2026-001
- Spec: 见 #12
- Design: 见 #13

### 实现
- [x] LoginForm 组件
- [x] useLogin hook
- [ ] API 集成

### 验收
- [x] 单元测试通过
- [ ] 集成测试通过
```
```

## 四、codebuddy-craft 集成

### 4.1 与现有架构的结合

```
┌─────────────────────────────────────────────────────────────┐
│                  codebuddy-craft SDD 集成                   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   01-Sources/规范收集/          02-Knowledge/提炼           │
│   ┌──────────────────┐       ┌──────────────────┐           │
│   │ 大厂编程规范     │  →   │ AI行为规范       │           │
│   │ SDD规范驱动开发  │       │ SDD应用指南      │           │
│   └──────────────────┘       └──────────────────┘           │
│                                    ↓                        │
│   03-Templates/               rules/                        │
│   ┌──────────────────┐       ┌──────────────────┐           │
│   │ SDD文档模板      │  →   │ agents.md        │           │
│   │                   │       │ 规范执行规则     │           │
│   └──────────────────┘       └──────────────────┘           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 实施建议

1. **从小型项目开始**：先在单个功能模块中实践 SDD
2. **建立规范库**：将大厂编程规范转化为 AI 可用的提示词
3. **强制执行**：通过 [[rules/common/agents.md]] 约束 AI 必须遵循规范
4. **持续迭代**：根据实践中遇到的问题持续优化规范

### 4.3 SDD 适用场景

| 场景 | 推荐度 | 说明 |
|-----|--------|------|
| 新功能开发 | ⭐⭐⭐⭐⭐ | 完整的 SDD 流程 |
| 重构项目 | ⭐⭐⭐⭐⭐ | 需要详细的变更追踪 |
| Bug 修复 | ⭐⭐⭐ | 可以简化流程 |
| 小改动 | ⭐⭐ | 流程过于繁琐 |
