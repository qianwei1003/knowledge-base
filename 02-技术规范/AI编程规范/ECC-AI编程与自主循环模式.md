## 1. AI-First Engineering（AI 优先工程）

AI 优先工程将 AI 视为核心开发参与者，强调规划质量和评估覆盖率，而非打字速度。代码评审从语法检查转向系统行为验证。

**核心要点：**

- 规划质量 > 打字速度；评估覆盖率 > 主观信心
- 架构必须对 AI 友好：明确边界、稳定契约、类型化接口、确定性测试
- 评审重点：行为回归、安全假设、数据完整性、故障处理、发布安全
- 工程师特征：清晰分解模糊工作、定义可衡量验收标准、生成高价值提示

**测试标准：** 对 AI 生成代码要求更高回归覆盖率 + 明确边界断言。

## 2. Agentic Engineering（智能体工程）

通过"评估优先循环"驱动开发：定义能力评估 + 回归评估 → 运行基线 → 实施 → 重新评估，形成闭环优化。

**核心要点：**

- **任务分解**：15 分钟单元规则——可独立验证、主要风险、清晰完成条件
- **模型路由**：Haiku（分类/样板）→ Sonnet（实施/重构）→ Opus（架构/根因/多文件）
- **会话策略**：紧密耦合继续同会话；阶段转换后开新会话；里程碑后压缩而非调试中
- **AI 代码审查重点**：不变量/边界、错误边界、安全假设、隐藏耦合/上线风险
- **成本纪律**：跟踪模型/令牌/重试/实际用时/成功失败

## 3. Autonomous Loops（自主循环模式谱系）

| 模式 | 复杂度 | 最适合场景 |
|------|--------|------------|
| 顺序管道（`claude -p`） | 低 | 日常开发、脚本化 |
| NanoClaw REPL | 低 | 交互式持久会话 |
| 无限智能体循环 | 中 | 并行内容生成 |
| 持续 Claude PR 循环 | 中 | 跨天迭代项目 |
| 去草率化模式 | 附加 | 任何实现步骤后 |
| Ralphinho RFC-DAG | 高 | 多单元并行 |

**各模式关键机制：**

- **顺序管道**：`set -e` 退出传播、步骤隔离、避免否定指令
- **NanoClaw**：会话历史持久化到 `~/.claude/claw/{session}.md`
- **无限智能体**：编排器分配创意方向（防止重复）、波次管理（3-5 个/波）
- **持续 Claude PR**：`SHARED_TASK_NOTES.md` 桥接迭代上下文、CI 失败自动修复、完成信号机制
- **去草率化**：单独清理步骤（不要负面指令）、移除过度测试/防御代码
- **Ralphinho DAG**：RFC 分解 → 层级管道（trivial→large 不同深度）→ 合并队列驱逐

## 4. Continuous Agent Loop（持续代理循环）

持续代理循环通过自动化的检查点机制保障项目推进，支持跨天迭代。

**循环选择决策：**

```
strict CI      → continuous-pr
RFC 分解       → rfc-dag
并行内容生成   → infinite
默认           → sequential
```

**推荐栈：** RFC 分解 + 质量门 + 评估 + 会话持久化

**故障模式：** 空转、重复重试、队列停滞、成本漂移

**恢复流程：** 冻结 → `/harness-audit` → 缩小范围 → 重置标准

## 5. Eval Harness（评估驱动开发 EDD）

评估驱动开发将评估视为 AI 开发的"单元测试"，通过 pass@k 指标量化能力。

**核心理念：** 评估 = AI 开发的单元测试

**评分器类型：**

- 代码评分器（确定性）
- 模型评分器（LLM 评判）
- 人工评分器

**pass@k 指标：**

- `pass@1`：首次尝试成功
- `pass@3`：3 次内至少一次成功
- `pass^k`：k 次内全部成功（pass@3 > 90% 为目标）

**工作流：** 定义（编码前）→ 实现 → 评估 → 报告

**评估反模式：** 过度拟合提示、只测正常路径、追求通过率忽略成本

## 6. Coding Standards（编码标准）

**核心原则：** 可读性优先、KISS、DRY、YAGNI

**TypeScript 规范：**

```typescript
// 不可变性：spread 而非 mutation
const updated = { ...state, count: state.count + 1 };

// Async/Await 并行
const [users, orders] = await Promise.all([getUsers(), getOrders()]);

// Zod Schema 验证输入
import { z } from 'zod';
const UserSchema = z.object({ id: z.string(), name: z.string() });
```

**React 规范：**

```typescript
// 组件 = 函数 + 类型
type Props = { initialCount: number };
function Counter({ initialCount }: Props) {
  const [count, setCount] = useState(initialCount);
  // 状态更新用函数式
  setCount(prev => prev + 1);
  return <div>{count}</div>;
}
```

**API 设计：** 统一响应格式 `ApiResponse<T>`、REST 约定、Zod 输入验证

**代码异味：** 长函数（>50行）、深层嵌套（>4层）、魔法数字

**测试 AAA：** Arrange-Act-Assert、描述性测试名

## 7. Verification Loop（验证循环）

六阶段验证确保每次交付的质量底线。

**六阶段流程：**
构建 → 类型检查 → 代码规范 → 测试套件 → 安全扫描 → 差异审查

**报告格式：**

```
构建: ✅ | 类型检查: ✅ | Lint: ✅ | 测试: ✅ | 安全: ✅ | 差异: ⚠️
总体: 5/6 通过
待修复: [diff] 列表
```

**触发时机：** 每 15 分钟或重大变更后运行 `/verify`

## 8. TDD Workflow（TDD）

测试驱动开发，通过快速红绿循环保证代码质量。

**AAA 工作流：**
用户旅程 → 测试用例 → RED（失败）→ GREEN（通过）→ 重构 → 覆盖率验证

**覆盖率要求：** 80%+（单元 + 集成 + E2E）

**测试类型：**

- 单元测试：Jest
- 集成测试：`NextRequest` 测试 API
- E2E 测试：Playwright

**常见错误：**

- 测试实现细节（应测试行为）
- 脆弱定位器（用语义定位器）
- 测试无隔离

## 9. Strategic Compact（战略精简）

战略精简通过有意识地压缩上下文来控制令牌成本，同时保持关键信息不丢失。

**何时压缩：**

- ✅ 研究 → 规划后
- ✅ 里程碑后
- ✅ 调试后
- ❌ 实施中不压缩

**令牌优化策略：**

- 触发表惰性加载（按需加载 skill）
- 上下文组合感知（`CLAUDE.md` 始终加载）
- `COMPACT_THRESHOLD` 环境变量控制提醒频率

## 10. Prompt Optimizer（提示优化器）

六阶段提示优化，将用户意图精准映射到执行方案。

**六阶段流程：**
项目检测 → 意图检测 → 范围评估 → ECC 组件匹配 → 缺失上下文 → 工作流推荐

**意图分类：**
新功能 / 错误修复 / 重构 / 研究 / 测试 / 审查 / 文档 / 基础设施 / 设计

**范围阈值：**

- 微小（<50行）→ Haiku
- 中等 → Sonnet
- 史诗级（多会话/多 PR）→ Opus

**ECC 组件映射表：** 意图 + 技术栈 → 命令/技能/代理

## 11. Planner（规划专家）

系统化规划，将模糊需求转化为可执行的阶段分解。

**规划格式：**

```
概述 → 需求 → 架构变更 → 实施步骤 → 测试策略 → 风险/缓解 → 成功标准
```

**阶段分解要求：**
每个步骤包含：文件路径 + 操作 + 原因 + 依赖 + 风险

**规模划分：** MVP → 核心体验 → 边界情况 → 优化

**危险信号（需检查）：**

- >50行函数
- >4层嵌套
- 重复代码
- 缺少错误处理

---

## 12. Loop Operator（循环操作员）

循环操作员负责安全运行自主循环，监控系统状态并在异常时升级。

**核心职责：**

- 安全运行自主循环
- 跟踪进度检查点
- 检测停滞和重试风暴

**必要检查：**

- 质量门激活
- 评估基线存在
- 回滚路径存在
- 分支隔离

**升级条件（满足任一即升级）：**

- 连续两检查点无进展
- 相同堆栈跟踪重复失败
- 成本漂移超预算
- 合并冲突
