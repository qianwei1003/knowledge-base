## 核心原则

1. **零依赖**：默认一个自包含的 HTML 文件，内联 CSS 和 JS
2. **视口适配强制**：每个幻灯片必须适配一个视口，无内部滚动
3. **先展示，不说教**：用视觉预览代替抽象风格问卷
4. **独特设计**：避免通用紫色渐变、Inter 字体白底、模板感
5. **生产质量**：保持代码注释完善、可访问、响应式、高性能

## 工作流

### Step 1: 检测模式

- **新建演示文稿**：用户有主题、笔记或完整草稿
- **PPT 转换**：用户有 .ppt 或 .pptx 文件
- **增强**：用户已有 HTML 幻灯片，想要改进

### Step 2: 发现内容

只问最小必要：

- 用途：pitch、教学、会议演讲、内部更新
- 时长：短（5-10）、中（10-20）、长（20+）
- 内容状态：已完成的文案、粗略笔记、只有主题

### Step 3: 发现风格

默认视觉探索：
1. 问用户 deck 应该产生什么感受：印象深刻、充满活力、专注、启发
2. 生成 3 个单页预览文件在 `.ecc-design/slide-previews/` 下
3. 每个预览必须自包含，清晰展示字体/颜色/动效，控制在约 100 行内
4. 让用户选择或混合元素

### Step 4: 构建演示文稿

输出：
- `presentation.html` 或 `[名称].html`

必须包含：
- 语义化幻灯片 section
- 视口安全 CSS 基线
- CSS 自定义属性（主题变量）
- 键盘、滚轮、触摸导航控制器
- Intersection Observer 揭示动画
- reduced-motion 支持

### Step 5: 强制视口适配

作为硬门槛：

- 每个 `.slide` 必须使用 `height: 100vh; height: 100dvh; overflow: hidden;`
- 所有文字和间距必须用 `clamp()` 缩放
- 内容不fit时，分成多个幻灯片
- 永远不要让文字缩小到不可读以下
- 永远不要在幻灯片内出现滚动条

### Step 6: 验证

检查以下尺寸：
- 1920x1080
- 1280x720
- 768x1024
- 375x667
- 667x375

## 内容密度限制

| 幻灯片类型 | 限制 |
|------------|------|
| 标题 | 1 个标题 + 1 个副标题 + 可选 tagline |
| 内容 | 1 个标题 + 4-6 个要点或 2 段短文 |
| 特性网格 | 最多 6 个卡片 |
| 代码 | 最多 8-10 行 |
| 引言 | 1 个引言 + 归因 |
| 图片 | 1 张图片，受视口约束 |

---

## PPT/PPTX 转换

1. 优先用 `python3` + `python-pptx` 提取文本、图片和笔记
2. 如果 `python-pptx` 不可用，询问是否安装或回退到手动导出流程
3. 保留幻灯片顺序、演讲者笔记和提取的资产
4. 提取后运行与新演示文稿相同的选择风格工作流

## 反模式

- 无视觉辨识度的通用创业渐变
- 除非故意编辑风格否则使用系统字体
- 长篇要点墙
- 需要滚动的代码块
- 短屏幕上破坏的固定高度内容框
- 无效的否定 CSS 函数如 `-clamp(...)`

---

## 交付检查清单

- [ ] 演示文稿可在浏览器中本地运行
- [ ] 每个幻灯片适配视口，无滚动
- [ ] 风格独特且有意图
- [ ] 动效有意义，不嘈杂
- [ ] 支持 reduced motion
- [ ] 在交付时解释文件路径、使用的预设、幻灯片数量和主题定制点

---

**记住**：独特性优于安全性。一个有明确视觉观点的演示文稿，胜过十个看起来像模板的 deck。
