# GitHub 工程团队：Prompt Engineering 与 LLM 应用开发指南

> 来源：GitHub Engineering Blog (2023-07-17)
> 作者：GitHub 工程团队
> 原始链接：https://github.blog/engineering/prompt-engineering-guide-generative-ai-llms/

## LLM 基础：在 1600 个 Token 内理解一切

LLM 本质上在做的事情很简单：预测下一个最有可能出现的 Token（字母组）。

- 你的手机输入法也会预测下一个词——LLM 就是在大规模上做同样的事
- 但 LLM 的训练基于"公共域中已知的所有文档"，因此几乎能表现出"常识"
- LLM 有时会自信地编造不真实的信息——称为"幻觉（Hallucinations）"

## 构建 LLM 应用的核心：域映射

**所有使用 LLM 的应用实际上都在两个域之间做映射**：用户域（User Domain）↔ 文档域（Document Domain）。

### 示例：自动化客服助手

用户的自然语言请求 → 转换为"伪文档" → LLM 补全文档 → 转换回用户的语言

关键步骤：
1. **转录**用户语音为文本
2. **构建文档上下文**：设定文档类型（如"ISP IT Support Transcript"）
3. **注入相关知识**：搜索相关文档并自然地编织到"伪文档"中
4. **LLM 补全**：基于完整上下文生成响应
5. **转换回用户域**：文本转语音或其他输出形式

> 💡 **这本质上就是 Prompt Engineering**：构建足够的上下文让 AI 产生最佳输出。

## GitHub Copilot 的 Prompt Engineering 管线

### 核心挑战

GitHub 上的文件通常是**已完成**的（编译通过），但用户在输入时代码通常是**未完成**的。仅仅预测光标后的文本是一种浪费——忽略了大量可用的上下文。

### 步骤一：收集上下文

Copilot 运行在 IDE 中，能利用 IDE 提供的一切信息——但**速度至关重要**：

- **每额外 10ms**，建议及时到达的概率**降低约 1%**

可用的上下文包括：
- **文件元数据**：语言、文件名（通过扩展名暗示语言并定调）
- **打开的标签页**：IDE 中打开的其他仓库文件（最多 20 个最近的）
- **光标前后代码**：当前文件光标前后的内容
- **其他仓库文件**：通过相似性和引用关系选择
- **最近编辑**：其他文件中的最近更改

### 步骤二：构建 Prompt

将收集的上下文编织成一个 LLM 可以理解的"文档"，核心问题始终是：**模型要完成的是什么类型的文档？**

### 步骤三：生成与后处理

LLM 返回原始补全后，需要进行后处理：
- 确保补全与用户已有的输入不重叠
- 验证语法合法性
- 可能生成多个建议供用户选择

## Prompt Engineering 的通用模式

### 1. 上下文工程（Context Engineering）

- 收集所有可能相关的上下文
- 在速度和上下文量之间做权衡
- 优先使用"强信号"上下文（如文件名、导入语句、打开的标签页）

### 2. 角色设定

通过设定角色来引导 LLM 的行为：
```
你是一位 IT 支持专家，擅长解决网络连接问题
```

### 3. 知识注入

当 LLM 的内置知识不足时：
- 搜索相关知识文档
- 将知识自然地嵌入到"伪文档"中
- 引导 LLM 利用注入的知识回答问题

### 4. 迭代优化

- Prompt Engineering 是一个持续迭代的过程
- 需要不断测试和优化 Prompt 的效果
- 监控模型的输出质量并调整策略

## 关键洞察

1. **上下文比 Prompt 文本更重要**：好的上下文可以让简单的 Prompt 产生好的结果
2. **速度是关键**：在交互式环境中，延迟直接影响用户体验
3. **多模态上下文**：软件开发是深度互联、多模态的挑战，利用越多复杂性，输出质量越好
4. **Prompt Engineering 是工程而非魔法**：有可复现的模式和管线

## 适用范围

这些模式不仅适用于代码补全，也适用于：
- 对话式搜索
- 写作助手
- 自动化 IT 支持
- 以及任何基于 LLM 的应用

> **核心原则：任何使用 LLM 的应用都在做域映射。Prompt Engineering 就是优化这个映射的艺术和科学。**
