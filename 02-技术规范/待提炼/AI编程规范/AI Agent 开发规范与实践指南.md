# 如何从零到一写好一个 AI Agent

> 来源：CSDN博客 https://blog.csdn.net/QingMengZ/article/details/159510215
> 整理时间：2026-04-19

---

## 1. 优秀 Agent 的评判标准

一个"好"的 Agent 应当具备以下特征：

- **目标聚焦**：不偏题，不胡言乱语
- **工具鲁棒性**：调用工具失败时能自我恢复，而不是直接崩溃
- **边界清晰**：知道自己能做什么，更要知道自己不能做什么（避免危险操作）
- **可观测性**：开发者能清晰看到它的思考过程（Thought Process）

## 2. 核心架构设计规范

### 2.1 打造强大的"大脑"（Prompt 工程）

写 Agent 的 System Prompt 与写普通对话的 Prompt 完全不同，需要定义"系统级行为"：

1. **角色设定（Persona）**：清晰定义它的身份和目标
2. **行为准则（Instructions）**：用编号列出严格的规则
   - 例如："1. 永远先搜索再回答；2. 如果工具返回 Error，尝试更换参数重试"
3. **输出格式（Format）**：强制要求 LLM 输出 JSON 或指定的 XML 格式，以便代码精准解析

> 💡 **Tip**：尽量使用能力最强的模型（如 GPT-4o、Claude 3.5 Sonnet）作为核心路由大脑，开源小模型可用于执行具体的小任务。

### 2.2 工具设计法则（Tools Best Practices）

工具是 Agent 与物理世界交互的触手。工具写得不好，Agent 就会变成"瞎子和聋子"。

**核心原则**：

1. **原子化设计**：一个工具只做一件事。不要写一个既能搜索天气又能订机票的庞大 API
2. **清晰的描述（Docstring）**：LLM 是通过阅读工具的文字描述来决定是否调用的
3. **参数类型明确**：不要用 `Any`，用 `str | int | list[str]` 等明确类型
4. **幂等性**：同一个工具被连续调用两次，不应产生副作用（除非是写操作）
5. **错误信息可读**：返回结构化错误，不要返回一串 stack trace

```python
# ❌ 坏示例
@tool
def search(query: str) -> str:
    """搜索"""
    ...

# ✅ 好示例
@tool
def web_search(query: str, max_results: int = 5) -> list[dict]:
    """
    使用搜索引擎查询互联网信息。

    Args:
        query: 搜索关键词，2-6个词最佳
        max_results: 返回结果数量，默认5，最大20

    Returns:
        包含 title, url, snippet 字段的字典列表
    """
    ...
```

### 2.3 记忆系统设计

Agent 的记忆分为三层：

| 层级 | 类型 | 实现 | 生命周期 |
|------|------|------|---------|
| L1 | 工作记忆（Working Memory） | 当前对话上下文 | 单次会话 |
| L2 | 短期记忆（Short-term） | 会话摘要 / 压缩 | 跨轮次 |
| L3 | 长期记忆（Long-term） | 向量数据库 / 文件 | 跨会话 |

**关键设计原则**：

1. **不要把所有东西塞进上下文**：上下文窗口有限，需要智能摘要
2. **分层检索**：先搜长期记忆，再补充短期记忆，最后放工作记忆
3. **遗忘机制**：过时的信息要能被标记和清理

### 2.4 规划与执行策略

主流的 Agent 规划策略：

| 策略 | 特点 | 适用场景 |
|------|------|---------|
| **ReAct** | 推理-行动交替，逐步迭代 | 动态环境、需实时纠偏 |
| **Plan-and-Execute** | 先规划全局步骤，再逐步执行 | 步骤明确的长期复杂任务 |
| **Reflection** | 执行后自省，迭代改进 | 需要高质量输出的任务 |

**最佳实践**：两者并非互斥，可结合使用——规划阶段采用 CoT 生成全局步骤，执行阶段在每个步骤内嵌入 ReAct 子循环。

## 3. 2026 年 Agent 生态关键技术

### 3.1 A2A 协议（Agent-to-Agent）

Google 主导的协议（2025年4月开源，现由 Linux Foundation 治理），让不同平台的 Agent 能互相协作：

- 每个 Agent 发布 **Agent Card**（数字名片）：包含名称、Skills、端点 URL
- 需要帮忙的 Agent 通过发现机制找到合适伙伴
- 通过 A2A 协议发送任务、接收结果

### 3.2 MCP（Model Context Protocol）

标准化工具调用协议，解决"每个框架有自己的工具格式"的问题：

- 统一的工具注册与调用标准
- 支持 SSE、Stdio 等多种传输方式
- 主流框架（Claude、OpenAI、LangChain）均已支持

### 3.3 Agent Skills

标准化的能力模块化方案：

- 核心文件：`SKILL.md`（定义任务执行流程和规则）
- 附加目录：`scripts/`（可执行脚本）、`references/`（参考文档）、`assets/`（附件素材）
- 采用渐进式披露架构，模型仅在推理时动态加载必要规则片段

## 4. 多 Agent 协作架构

2026 年主流架构不再是"一个万能 Agent"，而是角色分工：

| 角色 | 职责 |
|------|------|
| Planner | 任务拆解 |
| Executor | 调工具执行 |
| Critic | 质量校验 |
| Memory | 状态管理 |

### 设计原则

1. **单一职责**：每个 Agent 只负责一个领域
2. **明确接口**：Agent 间通过结构化消息通信
3. **可插拔**：可以替换某个 Agent 而不影响整体
4. **容错**：某个 Agent 失败时，系统应有 fallback

## 5. 安全与边界

### 5.1 必须设置的安全规则

```python
SAFETY_RULES = [
    "禁止执行任何删除操作（rm、DROP TABLE 等），除非用户明确确认",
    "禁止访问用户的敏感文件（如 .env、id_rsa）",
    "禁止向外部服务发送用户私有数据",
    "工具调用失败时，不要重试超过 3 次",
    "如果不确定操作的安全性，先问用户",
]
```

### 5.2 人机协作模式

- **Autonomous**：Agent 自主执行，适合低风险任务
- **Supervised**：Agent 执行前需用户确认，适合中风险任务
- **Assisted**：Agent 只建议不执行，适合高风险任务

## 6. 实战检查清单

### 开发前

- [ ] 明确 Agent 的目标边界（做什么 / 不做什么）
- [ ] 列出所有需要的工具及其参数
- [ ] 设计记忆系统（哪些需要持久化）
- [ ] 确定安全级别和需要的人机协作模式

### 开发中

- [ ] System Prompt 是否足够具体？
- [ ] 工具描述是否让 LLM 能准确选择？
- [ ] 错误处理是否完善（工具失败、API 超时等）？
- [ ] 是否有防止无限循环的机制（最大步数限制）？

### 上线前

- [ ] 压力测试：长时间运行是否稳定？
- [ ] 边界测试：超出能力范围时是否优雅降级？
- [ ] 安全审计：是否有越权风险？
- [ ] 可观测性：能否追踪每一步的决策过程？

---

## 7. 总结

写好一个 AI Agent 的核心心法：

1. **Prompt 是灵魂**：花 80% 的时间打磨 System Prompt
2. **工具是手脚**：原子化、描述清晰、错误友好
3. **记忆是底蕴**：分层设计、智能摘要、定期清理
4. **安全是底线**：永远设置边界、永远允许人工介入
5. **可观测是保障**：每一步决策都应可追溯
