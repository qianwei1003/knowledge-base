# Windsurf 全局规则与项目规则实战配置指南

> 来源：CSDN 博客 | 作者：hp777 | 发布时间：2026-02-13
> 原文链接：https://blog.csdn.net/hp777/article/details/155327193

## 一、为什么需要规则？从"无效对话"到"精准输出"

没有规则的 AI 辅助 = 让不了解代码库的新人直接改核心逻辑。配置规则后，AI 生成的代码从第一行就大概率符合预期。

规则本质 = 给 AI 提前准备"共享上下文"（项目入职手册 + 个人编程习惯说明书）。

## 二、全局规则（Global Rules）——"编程宪法"

### 文件位置
`~/.config/Codeium/global_rules.md`（因系统而异）

最方便的打开方式：编辑器右下角 Windsurf-Setting → Set Global Rules

### 全局规则框架模板

```markdown
# 全局 AI 助手规则

## 角色与目标
你是一位拥有10年以上全栈开发经验的高级工程师，尤其精通现代前端技术栈（Vue/React、TypeScript、Node.js）。
你的任务是作为我的编程伙伴，高效、准确地协助我完成开发工作。你的回答应直接、务实，以解决问题为首要目标。

## 核心交互原则
1. **主动思考，而非被动应答**：在动手写代码前，先分析需求，如果发现模糊、矛盾或缺失的信息，必须主动向我提问确认，而不是基于假设生成可能错误的代码。
2. **代码一致性优先**：生成的任何代码，在风格、命名约定、技术选型上，必须与当前活动文件及项目中的现有代码保持高度一致。如果文件是 Vue 3 的 `<script setup>`，就绝不用 Options API。
3. **修改范围控制**：严格限定修改我要求的范围，不要自作主张改无关代码。
```

### 关键原则

1. **不要复制别人的规则**，根据自己痛点定制
2. **角色定义**要具体（技术栈、经验年限、工作风格）
3. **交互原则**要明确（主动提问 vs 直接执行、一致性要求、范围控制）

## 三、项目规则（Project Rules）——"项目入职手册"

### 放置方式
- `.windsurfrules` 文件（放在项目根目录）
- 或在 `.windsurf/` 目录下配置

### 项目规则应包含

```markdown
# 项目规则

## 技术栈
- 框架：Vue 3 + Composition API (`<script setup>`)
- 状态管理：Pinia
- 样式方案：Tailwind CSS
- 构建工具：Vite

## 代码规范
- 使用 TypeScript，禁止 any 类型
- 组件命名：PascalCase
- 文件命名：kebab-case
- 优先使用 Composition API + `<script setup>`
- 使用 storeToRefs 解构 Pinia store

## 项目结构约定
- 组件放在 src/components/
- composables 放在 src/composables/
- API 请求放在 src/api/
- 类型定义放在 src/types/

## 禁止事项
- ❌ 禁止使用 Vue 2 Options API
- ❌ 禁止直接修改 props
- ❌ 禁止在组件中直接调用 API（通过 composable 封装）
- ❌ 禁止使用 this 关键字
```

## 四、规则配置最佳实践

### 1. 分层设计
- **全局规则**：个人技术偏好、通用原则（对所有项目生效）
- **项目规则**：特定项目的技术栈、架构约定、文件结构

### 2. 渐进式完善
- 不必一开始就写完美的规则
- 从频繁纠正 AI 的场景中提取规则
- 随着使用积累逐步完善

### 3. 规则要可执行
- ❌ "写好代码"（太模糊）
- ✅ "使用 TypeScript，禁止 any 类型；组件使用 PascalCase 命名；优先 `<script setup>`"

### 4. 定期审查
- 项目技术栈变更时同步更新规则
- 发现 AI 重复犯同类错误时补充规则

## 五、常见问题

**Q: 全局规则和项目规则冲突怎么办？**
A: 项目规则优先级更高，会覆盖全局规则中的冲突项。

**Q: 规则写多长合适？**
A: 全局规则控制在 500 字以内（AI 注意力有限），项目规则可以更详细但要结构化。

**Q: AI 不遵守规则怎么办？**
A: 检查规则是否足够具体，必要时在对话中再次强调关键约束。

## 关键启示

1. **规则 = 共享上下文**，本质是减少每次对话的沟通成本
2. **分层设计**：全局管偏好，项目管约定
3. **从痛点出发**：记录 AI 常犯的错误，反向提炼规则
4. **具体 > 抽象**：用明确的可执行指令替代模糊描述
