# Prompt Engineering 最佳实践 — DigitalOcean 完整指南

> 来源：DigitalOcean 官方技术文章 (2025-12)
> 本文系统介绍了 Prompt Engineering 的核心概念、8种技术类型和10条最佳实践，适用于所有 LLM 用户。

## 一、什么是 Prompt Engineering

Prompt Engineering 是精心设计输入以从大语言模型获取期望输出的过程。包括：
- 用词选择
- 指令排序
- 上下文提供
- 示例包含

高级层面还包括：知道何时使用 few-shot prompting、chain-of-thought prompting 等技术。

## 二、为什么重要

- **减少幻觉、提高准确性** — 明确允许模型说"我不知道"
- **提升代码生成质量** — 指定编程语言、输入输出、边界情况
- **使复杂推理更透明可靠** — Chain-of-thought 让推理过程可审查
- **控制输出格式和结构** — Few-shot 教模型精确的样式和结构
- **改善多模态任务效果** — 图像/视觉模型同样受提示词质量影响

## 三、8种 Prompt Engineering 技术

### 1. Zero-shot Prompting

直接给指令，不提供示例。适合简单、明确格式显而易见的任务。

**适用场景**："翻译这句话到法语"、"用两句话总结这段文字"

### 2. Few-shot Prompting

提供1个或多个输入-输出示例，帮助模型学习你期望的模式。

**适用场景**：需要特定格式、风格或约定的输出

**技巧**：从一个示例（one-shot）开始，如果输出还不符合预期再增加。

### 3. Chain-of-thought (CoT) Prompting

让模型在给出最终答案前展示推理过程。最简单的版本是加上"Think step-by-step"。

**适用场景**：数学问题、逻辑推理、多步分析

**优势**：中间步骤出错时可以及时发现，而不是等最终答案出错。

### 4. Tree-of-thought (ToT) Prompting

同时探索多条推理路径，评估每条路径，必要时回溯。

**适用场景**：需要策略规划的复杂问题（游戏、多步决策）

**数据支撑**：Princeton & Google DeepMind 研究显示，ToT 在 Game of 24 基准上达到 74% 成功率，远超 CoT (49%) 和标准 prompting (33%)，但单次成本更高 ($0.74 vs $0.47 vs $0.13)。

### 5. Role/Persona Prompting

给模型分配特定身份或专业知识。

**技巧**：现代模型足够复杂，过于强烈的角色分配可能反而限制帮助性。更轻量的方式是明确你想要的视角，而非分配角色。

**推荐**：用"从风险容忍和长期增长的角度分析这个投资组合"代替"你是一个世界级专家"。

### 6. Prompt Chaining

将复杂任务分解为顺序步骤，前一步的输出成为下一步的输入。

**适用场景**：分析→综合的工作流，如：总结文档 → 评估总结完整性 → 根据反馈改进

**实现方式**：API 编程实现或手动在聊天中复制输出

### 7. Meta-prompting

用 LLM 生成、优化或改进 prompt，而非手动编写。

**适用场景**：需要可复用模板、系统性优化 prompt、大规模应用中 prompt 质量直接影响结果

**工具示例**：ChatPRD — 将高层产品概念转为结构化 prompt

### 8. Reflection Prompting

让模型在给出初始回答后，自我审查、评估并改进。

**适用场景**：准确性要求高的任务

**示例**："Review the code above for bugs or edge cases, then provide a corrected version."

**权衡**：增加延迟，但常能提升复杂推理和写作任务的质量。

## 四、10条 Prompt Engineering 最佳实践

### 1. 尽可能具体

- 提供详细的上下文
- 明确输出格式、长度、语气
- 包含所有相关约束条件

### 2. 提供清晰的上下文和背景

- 给出任务背景
- 说明你的角色和受众
- 提供相关数据或参考资料

### 3. 使用示例（Few-shot）

- 展示你期望的输入-输出模式
- 2-3个示例通常足够
- 示例应覆盖你关心的变化维度

### 4. 定义期望的输出格式

- 指定 JSON、表格、列表等格式
- 明确各字段要求
- 提供格式模板

### 5. 告诉模型该做什么，而非不该做什么

- 用"使用正式语气"代替"不要用俚语"
- 用"给出3个选项"代替"不要只给一个答案"
- 正面指令比负面约束更有效

### 6. 分解复杂任务

- 大任务拆为小步骤
- 使用 Prompt Chaining 逐步执行
- 每步聚焦一个明确目标

### 7. 迭代优化 Prompt

- 从简单 prompt 开始
- 根据输出调整
- 记录有效和无效的变体
- 系统化测试

### 8. 理解模型的局限性

- 模型会幻觉
- 训练数据有截止日期
- 长上下文中间的信息容易被忽略
- 不要假设模型知道你的内部信息

### 9. 测试和验证

- 在不同输入上测试 prompt
- 检查边界情况
- 评估一致性和可靠性
- 量化效果改进

### 10. 保持 Prompt 简洁但有信息量

- 避免不必要的重复
- 删除冗余指令
- 每条指令都应有明确目的
- 信息密度 > 长度

## 五、技术对比总结

| 技术 | 复杂度 | 适用场景 | 成本 |
|------|--------|----------|------|
| Zero-shot | 低 | 简单明确任务 | 低 |
| Few-shot | 低-中 | 格式/风格要求 | 低-中 |
| CoT | 中 | 推理/数学/逻辑 | 中 |
| ToT | 高 | 策略规划/复杂决策 | 高 |
| Role | 低 | 领域特定任务 | 低 |
| Chaining | 中-高 | 多阶段工作流 | 中-高 |
| Meta | 中 | Prompt优化/模板 | 中 |
| Reflection | 中 | 高准确性要求 | 中 |

## 六、关键要点

1. **Prompt Engineering 的核心不是"技巧"，而是"清晰表达需求"**
2. **选择技术应根据任务复杂度，而非追求"高级"**
3. **迭代优化比一次性完美更重要**
4. **正面指令 > 负面约束**
5. **复杂任务必须分解，不要指望一个 prompt 解决所有问题**
