# OpenSpec 规范驱动开发实战指南

> 来源：2025年10月 CSDN 技术文章，介绍 OpenSpec 开源框架——一种专为 AI 编程助手设计的规范驱动开发方法。

---

## 一、什么是 OpenSpec

OpenSpec 是一种**规范驱动（spec-driven）**的开源开发框架，主要面向 AI 编程助手（如 Claude Code、GitHub Copilot、Cursor 等）而设计。

### 1.1 核心问题

在 AI 辅助开发中，常见痛点包括：

- **需求模糊**：AI 接收到模糊指令，生成偏离预期的代码
- **指令歧义**：同一需求不同人描述不同，AI 输出不一致
- **缺乏可追溯性**：AI 生成的代码难以追溯对应的需求来源
- **可维护性差**：AI 代码修改后偏离原始设计意图

### 1.2 核心理念

> **先写规范，再写代码** — 让规范成为人类和 AI 的共同真理来源（Single Source of Truth）。

---

## 二、核心工作流

OpenSpec 定义了一个三阶段闭环流程：

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  共识规范      │ ──→ │  AI 执行      │ ──→ │  自动验证      │
│  (Spec)       │     │  (Execute)   │     │  (Verify)    │
└──────────────┘     └──────────────┘     └──────────────┘
       ↑                                        │
       └────────────────────────────────────────┘
                    反馈修正
```

### 2.1 阶段一：共识规范（Spec）

由人类与 AI 共同撰写结构化的需求规范，包括：

| 规范要素 | 说明 | 示例 |
|---------|------|------|
| **功能描述** | 要实现什么 | "用户登录功能，支持邮箱和手机号" |
| **输入/输出** | 接口契约 | "输入：email(string), password(string)；输出：token(string)" |
| **边界条件** | 异常场景 | "密码长度 8-32 位，连续错误 5 次锁定" |
| **测试用例** | 验收标准 | "正确凭证返回 200，错误凭证返回 401" |
| **非功能需求** | 性能/安全约束 | "响应时间 < 200ms，密码 bcrypt 加密" |

### 2.2 阶段二：AI 执行

AI 根据规范自动生成代码、文档或变更提案。

**Prompt 模板示例：**
```
请根据以下规范生成代码：

## 功能描述
{description}

## 接口定义
- 输入：{inputs}
- 输出：{outputs}

## 边界条件
{edge_cases}

## 约束
- 语言：{language}
- 框架：{framework}
- 代码风格：遵循项目既有风格
```

### 2.3 阶段三：自动验证

生成的代码自动运行测试，验证是否符合规范：

- 单元测试覆盖所有测试用例
- 集成测试验证输入/输出契约
- 边界条件测试覆盖异常场景

## 三、规范文档结构

### 3.1 Spec 文件模板

```markdown
# {功能名称} 规范

## 概述
{一句话描述功能目的}

## 功能需求
### FR-001: {功能点1}
- **描述**：{详细说明}
- **优先级**：P0/P1/P2
- **验收标准**：
  - [ ] {标准1}
  - [ ] {标准2}

### FR-002: {功能点2}
...

## 接口定义
### API: {接口名称}
- **方法**：POST/GET/PUT/DELETE
- **路径**：/api/v1/{path}
- **请求体**：
  ```json
  {schema}
  ```
- **响应体**：
  ```json
  {schema}
  ```
- **错误码**：
  | 错误码 | 含义 | 处理建议 |
  |--------|------|---------|

## 非功能需求
- **性能**：{指标}
- **安全**：{约束}
- **兼容性**：{要求}

## 测试用例
| 编号 | 场景 | 输入 | 预期输出 | 优先级 |
|------|------|------|---------|--------|
| TC-001 | {场景} | {输入} | {输出} | P0 |

## 变更历史
| 日期 | 版本 | 变更内容 | 作者 |
|------|------|---------|------|
```

### 3.2 规范质量检查清单

- [ ] 功能描述无歧义（不同人理解一致）
- [ ] 每个功能点都有对应测试用例
- [ ] 边界条件完整覆盖
- [ ] 接口定义包含错误码
- [ ] 非功能需求可量化
- [ ] AI 可直接基于规范生成代码（无需额外口头说明）

## 四、SDD 与传统开发流程对比

| 维度 | 传统开发 | 规范驱动开发（SDD） |
|------|---------|-------------------|
| 需求传递 | 口头/文档 → 开发者 | 结构化规范 → AI + 开发者 |
| 代码生成 | 开发者手写 | AI 根据规范自动生成 |
| 质量保证 | 开发后测试 | 规范即测试用例 |
| 可追溯性 | 代码 ↔ 需求文档弱关联 | 规范 ↔ 代码强关联 |
| 迭代方式 | 改需求 → 改代码 | 更新规范 → AI 重新生成 |
| 协作效率 | 依赖人工沟通 | 规范作为共享契约 |

## 五、在 AI 编程工具中的应用

### 5.1 Claude Code

```
# 在 .claude/CLAUDE.md 或项目规范中引用

开发新功能时，请遵循 SDD 流程：
1. 先阅读 specs/ 目录下对应功能规范
2. 根据规范生成代码
3. 运行规范中定义的测试用例
4. 如果规范不明确，先提问确认，不要自行假设
```

### 5.2 Cursor

在 `.cursorrules` 中配置：
```
## 开发规范
- 所有新功能开发前，必须先确认对应 spec 文件存在
- 代码实现必须严格遵循 spec 中的接口定义
- 测试用例覆盖 spec 中定义的所有场景
- 代码变更需同步更新 spec 的变更历史
```

### 5.3 GitHub Copilot

在 Copilot Instructions 中设置：
```
When implementing new features:
1. Read the corresponding spec file in specs/ directory first
2. Generate code that strictly follows the spec's interface definitions
3. Include tests that cover all test cases defined in the spec
4. If the spec is ambiguous, ask for clarification before implementing
```

## 六、最佳实践

### 6.1 规范编写

1. **从小开始**：先为核心功能编写规范，逐步扩展
2. **迭代完善**：规范是活的文档，随着理解深入持续更新
3. **AI 参与**：让 AI 帮助完善规范的边界条件和测试用例
4. **团队共识**：规范需要团队评审，确保理解一致

### 6.2 代码生成

1. **一次一个功能**：不要一次给 AI 太多规范
2. **提供上下文**：告诉 AI 项目的技术栈和代码风格
3. **分步验证**：每生成一部分就验证，不要等到最后
4. **人工审查**：AI 生成的代码必须经过人工审查

### 6.3 常见陷阱

| 陷阱 | 说明 | 解决方案 |
|------|------|---------|
| 规范过于详细 | 限制 AI 的优化空间 | 定义"做什么"而非"怎么做" |
| 规范过于模糊 | AI 输出不符合预期 | 包含具体输入/输出示例 |
| 忽略边界条件 | AI 生成的代码缺少异常处理 | 规范必须包含边界条件和错误处理 |
| 规范与代码脱节 | 代码修改后规范未更新 | 规范变更纳入 CI 检查 |

## 七、与现有工具的集成

OpenSpec 可与现有 AI 编程工具链无缝集成：

- **Claude Code**：通过 CLAUDE.md 引用规范
- **Cursor**：通过 .cursorrules 配置规范驱动流程
- **GitHub Copilot**：通过 Copilot Instructions 引导
- **CI/CD**：规范变更触发自动代码生成和验证
- **PR Review**：代码是否符合规范作为 Review 标准

---

## 八、总结

OpenSpec 规范驱动开发（SDD）为 AI 辅助编程提供了一种结构化的方法论：

| 核心价值 | 说明 |
|---------|------|
| **降低歧义** | 结构化规范消除 AI 理解偏差 |
| **提升质量** | 规范即测试，自动验证代码正确性 |
| **增强可追溯性** | 代码 ↔ 规范强关联 |
| **团队协作** | 规范作为共享契约，减少沟通成本 |
| **人机协同** | 人类定义"做什么"，AI 实现"怎么做" |
