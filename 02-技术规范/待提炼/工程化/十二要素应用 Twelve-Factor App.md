# 十二要素应用 (The Twelve-Factor App)

> 来源：12factor.net（Heroku 官方方法论）https://12factor.net/zh_cn/
>
> 作者：Adam Wiggins（Heroku 联合创始人）
>
> 适用：任何语言和后端服务开发的 SaaS 应用

## 概述

12-Factor 为构建 SaaS 应用提供了方法论，核心理念：
- 使用标准化流程自动化配置
- 与操作系统最大限度解耦
- 适合现代云平台部署
- 将开发/预发布/线上环境差异降至最低
- 可在不改变工具/架构/流程的前提下扩展

---

## I. 基准代码（Codebase）

### 一份基准代码，多份部署

- 使用版本控制系统（Git/Mercurial/Subversion）
- 一份基准代码 = 一个应用
- 多个部署（生产、预发布、本地）共享同一代码，不同版本
- 多应用共享代码 → 拆分为共享库，通过依赖管理加载

**原则：** 一个代码库，多个部署

---

## II. 依赖（Dependencies）

### 显式声明依赖关系

- 通过**依赖清单**（如 Gemfile、package.json、requirements.txt）显式声明
- 通过**依赖隔离**确保不调用未声明的系统级依赖
- 声明和隔离必须同时使用
- 不隐式依赖系统工具（如 ImageMagick），应打包进应用

**原则：** 显式声明所有依赖，永远不要隐式依赖系统级类库

---

## III. 配置（Config）

### 在环境中存储配置

代码与配置严格分离。配置包括：
- 数据库、缓存等后端服务配置
- 第三方服务证书（如 AWS S3、Twitter API）
- 部署特有的配置（如域名）

**判断标准：** 代码能否立刻开源而不暴露敏感信息？

**推荐方式：** 环境变量（env vars）
- 跨部署修改无需改代码
- 几乎不可能意外签入代码库
- 语言/系统无关

**反对方式：**
- 代码中硬编码常量
- 配置文件签入版本控制

**原则：** 将配置存储在环境中，不要将敏感信息写入代码

---

## IV. 后端服务（Backing Services）

### 把后端服务当作附加资源

数据库、队列、缓存、SMTP 等都视为**附加资源**，通过 URL 或服务凭据从配置中获取。

- 本地 MySQL ↔ 第三方 Amazon RDS → 只需改配置中的 URL
- 每个不同服务 = 一个独立资源
- 资源与部署保持松耦合，可按需卸载/加载

**原则：** 把后端服务当作附加资源，通过 URL/凭据访问

---

## V. 构建，发布，运行（Build, Release, Run）

### 严格分离构建和运行

三个阶段严格分离：

| 阶段 | 说明 |
|------|------|
| **构建** | 代码 → 可执行包，编译依赖和资源文件 |
| **发布** | 构建结果 + 配置，可以立即运行 |
| **运行** | 启动进程，执行应用 |

- 每个发布版本有唯一 ID（时间戳或递增数字）
- 发布不可修改，任何变更 → 新版本
- 不直接修改运行中代码

**原则：** 严格分离构建、发布、运行三个阶段

---

## VI. 进程（Processes）

### 以无状态进程运行应用

- 应用进程必须**无状态**、**无共享**
- 任何持久化数据存储在后端服务（数据库、Redis 等）
- 进程内存/磁盘空间只作临时缓存
- 禁止使用"粘性 Session"（将 session 缓存在进程内存）

**正确方式：** Session 数据存 Redis/Memcached（带过期时间）

**原则：** 进程无状态，任何需要持久化的数据存储在后端服务

---

## VII. 端口绑定（Port Binding）

### 通过端口绑定来提供服务

- 应用完全自我加载，不依赖网络服务器（Apache/Nginx）
- 通过端口绑定提供服务，监听该端口的请求
- 端口绑定使应用成为其他应用的**后端服务**

**示例：** Tornado（Python）/ Thin（Ruby）/ Jetty（Java）

**原则：** 通过端口绑定提供服务，而非依赖运行时注入的 Web 服务器

---

## VIII. 并发（Concurrency）

### 通过进程模型进行扩展

- 进程是一等公民
- 将不同工作分配给不同**进程类型**
  - Web 进程：处理 HTTP 请求
  - Worker 进程：处理后台任务
- 不依赖线程（垂直扩展有瓶颈）
- 借助操作系统进程管理器（systemd/Foreman/云平台）

**原则：** 进程类型分类，不同类型处理不同工作，通过进程模型水平扩展

---

## IX. 易处理（Disposability）

### 快速启动和优雅终止

- 进程应追求**最小启动时间**
- 接收 SIGTERM → 优雅终止：
  - 停止监听端口，拒绝新请求
  - 完成已接收请求后退出
- Worker 进程：当前任务退回队列
- 任务应设计为**可重复执行**（幂等性）

**原则：** 最小启动时间 + 优雅终止，最大化健壮性

---

## X. 开发环境与线上环境等价（Dev/Prod Parity）

### 尽可能保持开发、预发布、线上环境相同

| 对比 | 传统应用 | 12-Factor |
|------|---------|-----------|
| 部署间隔 | 数周 | 几小时 |
| 人员 | 开发≠运维 | 相同 |
| 环境 | 差异大 | 尽量接近 |

**关键要求：**
- 后端服务开发/预发布/线上使用**相同版本的相同服务**（不用 SQLite 开发 + PostgreSQL 生产）
- 使用 Docker/Vagrant 等工具使本地与线上无限接近

**原则：** 开发环境与线上环境尽可能保持接近

---

## XI. 日志（Logs）

### 把日志当作事件流

- 日志 = 所有运行中进程和后端服务输出流的**时间顺序汇总**
- 应用不管理日志文件写入
- 进程直接输出到 **stdout**
- 运行环境截获输出流，发送给日志系统（Logplex/Fluentd/Splunk）

**原则：** 将日志视为事件流，输出到 stdout，由运行环境管理

---

## XII. 管理进程（Admin Processes）

### 后台管理任务当作一次性进程运行

管理任务示例：
- 数据库迁移（`migrate`、`db:migrate`）
- REPL 控制台（`rails console`、`python`）
- 一次性脚本

**要求：**
- 与常规进程使用**相同的代码和配置**
- 使用**相同的依赖隔离**技术
- 管理代码与应用程序代码一起发布

**原则：** 管理进程与常规进程同等对待，通过同一发布版本运行

---

## 总结速查表

| 要素 | 核心原则 |
|------|---------|
| I. 基准代码 | 一份代码库，多份部署 |
| II. 依赖 | 显式声明，完全隔离 |
| III. 配置 | 环境变量存储配置 |
| IV. 后端服务 | 视为附加资源 |
| V. 构建/发布/运行 | 三阶段严格分离 |
| VI. 进程 | 无状态，无共享 |
| VII. 端口绑定 | 自我加载提供服务 |
| VIII. 并发 | 进程类型分类，水平扩展 |
| IX. 易处理 | 最小启动 + 优雅终止 |
| X. Dev/Prod 等价 | 开发预发布线上环境一致 |
| XI. 日志 | 事件流 → stdout |
| XII. 管理进程 | 一次性进程 = 常规进程 |
